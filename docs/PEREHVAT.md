> Источник истины — полевой скрипт `tools/cross_operations_v2.py` (отлажен на железе,
> идентичен версии со шкафа) + замеры `docs/geometry.md` (08.06.2026).
> Это ПРОЕКТ интеграции в приложение. Любая реализация валидируется на железе
> (сухой прогон), мок перехват не ловит. Железные правила механики — в CLAUDE.md.

## 1. Зачем перехват

Полочка широкая — **32 см между прорезями**, платформа узкая — **16 см между замками**
(`geometry.md`). Один замок не может за один ход и захватить полку в стойке, и вынести её
на другую сторону платформы. Поэтому: захватываем одним замком → сдвигаем лоток на
`LOCK_DISTANCE` так, чтобы под прорезь встал ВТОРОЙ замок → **перехватываем** вторым →
довозим. Это и есть перехват.

Окно выдачи — в **переднем** ряду (`config.CABINET window = FRONT, x=1, y=9`). Значит:
- книга из **переднего** ряда → окно: перехват НЕ нужен (передний замок, прямая выдача).
- книга из **заднего** ряда → окно: перехват НУЖЕН (задний замок → перехват на передний).
- возврат в **задний** ряд: из окна (перёд) → перехват на задний → положить.

Сейчас в приложении (`mechanics/algorithms.py` `take_shelf`/`give_shelf`) — только
**одиночный замок** (`lock1` FRONT / `lock2` BACK), перехвата нет. → кросс-рядные операции
не работают; отлажено лишь front→front и back→back (как и говорил Роман).

## 2. Полевые константы (из cross_operations_v2.py — НЕ менять без железа)

| Константа | Значение | Смысл |
|---|---|---|
| `TRAY_STEP / TRAY_DIR` | GPIO 18 / 27 | DIR 0=FRONT, 1=BACK |
| `TRAY_EN1 / EN2` | 25 / 26 | LOW=работа (перед движением опустить) |
| `TRAY_FREQ` | 12000 Гц | частота шагов лотка |
| `ENDSTOP_FRONT / BACK` | 7 / 20 | концевики лотка (1=нажат); **GPIO20 шумит** |
| `LOCK_FRONT / REAR` | 12 / 13 | сервозамки |
| **`LOCK_DISTANCE`** | **12600 шагов** | ход лотка между точками перехвата |
| **`TRAY_CENTER`** | **11300 шагов** | центральная (парковочная) позиция лотка |
| `LOCK_GRAB_PWM` | 1200 | захват (язычок поднят, полка заперта) |
| `LOCK_RELEASE_PWM` | 500 | отпускание (язычок опущен, полка свободна) |

Замок: подать PWM (1200 grab / 500 release) → `sleep 0.5` → снять (`pigs s pin 0`).
Финальное отпускание в ячейку — `strong` (3× по 500, затем 0) для надёжности.
Подвод к концевику — двухэтапный (FAST 12000 → backoff 1500 → SLOW 1500), `sensor_stable` 5.

## 3. Валидированная последовательность (rear_to_front, 8 шагов)

Из `cross_operations_v2.py:rear_to_front_v2` — взять из ЗАДНЕГО ряда, вынести в ПЕРЕДНИЙ:

1. Лоток → BACK endstop (задний замок под прорезью полки)
2. **REAR lock GRAB** (захватили из задней стойки)
3. Лоток → `LOCK_DISTANCE`(12600) к FRONT (передний замок встаёт под прорезь)
4. **REAR lock RELEASE** (отпускаем задний)
5. **FRONT lock GRAB** (перехватили передним) ← перехват
6. Лоток → FRONT endstop (точно по концевику)
7. **FRONT lock RELEASE strong** (положили / готово к выдаче)
8. Лоток → CENTER (11300 к BACK)

`front_to_rear` — зеркально (BACK↔FRONT, DIR инвертирован).

## 4. Полная кросс-рядная операция = каретка(XY) + перехват(лоток)

`cross_operations_v2` — это перехват ТОЛЬКО лотком при неподвижной каретке (перекладывает
полку между передней и задней стойками одной колонки). Полная боевая операция комбинирует
XY-перемещения каретки и tray-перехват. Пример — **выдача книги из заднего ряда**:

1. Каретка → XY задней ячейки (`get_cell_position('BACK', x, y)` — уже из полевой калибровки, #98).
2. Захват полки на платформу задним замком (лоток→BACK endstop, REAR grab).
3. **Перехват** REAR→FRONT (лоток +12600 к FRONT, RELEASE rear, GRAB front).
4. Каретка → XY окна (FRONT, 1, 9).
5. Выдвинуть в окно, открыть шторки, `wait_for_user` (детект по RRU9816, #2).
6. Обратно: перехват FRONT→ (для заднего ряда) и развоз в исходную ячейку.

Возврат в задний ряд — обратный порядок (из окна → перехват на задний → положить в заднюю ячейку).

## 5. Что менять в коде (предложение — реализуем и валидируем на железе)

### 5.1 Новый tray-примитив (порт cross_operations_v2 в приложение)
В `hardware/motors.py` (или новый `hardware/tray_handoff.py`) — НЕинтерактивные async-методы,
без `input()`, с таймаутом на каждое движение и progress-callback:
- `tray_to_endstop(target: 'FRONT'|'BACK')` — двухэтапный подвод (FAST→backoff→SLOW, sensor_stable, glitch-фильтр 5000 мкс на GPIO20 — issue #41).
- `lock_grab(lock)` / `lock_release(lock, strong=False)` — PWM 1200/500 → 0 (уже есть в `servos`, свериться со значениями: сейчас servos open=0°/close=95°, а поле — PWM 1200/500; **выверить соответствие**).
- `tray_handoff(from_row, to_row)` — 8-шаговая последовательность из §3 (LOCK_DISTANCE, перехват).

### 5.2 Интеграция в business-флоу
- `take_shelf(row, x, y)` / `give_shelf`: если ячейка в ЗАДНЕМ ряду И цель — переднее окно
  (или наоборот) → вставить `tray_handoff`. Если same-row — текущий одиночный замок.
- Альтернатива (чище): отдельные методы `deliver_to_window(cell)` / `stow_from_window(cell)`,
  которые сами решают, нужен ли перехват по `cell.row` vs `window.row`.
- Прогресс по WS — переиспользовать `_emit_progress`/`_KioskProgress` (добавить шаги перехвата в шкалу).

### 5.3 Сверить два мира tray-движения
Сейчас в приложении лоток ходит по `grab_front/grab_back` (шаговые extend1/retract/extend2,
дефолтные и одинаковые — см. `docs/FLOWS.md`), а поле — по `tray_to_endstop` + `LOCK_DISTANCE`.
**Поле точнее** (по концевику, а не по счётчику шагов). Предложение: для перехвата брать
endstop-логику cross_operations_v2 как канон, `grab_*` параметры не использовать для кросс-ряда.

## 6. Риски / валидировать на железе (мок не ловит)
- ⚠️ GPIO20 (BACK endstop) шумит — нужен glitch-фильтр (issue #41), иначе ложный «нажат» в перехвате.
- ⚠️ Соответствие PWM замков: поле 1200/500, `servos.py` — углы 0/95. Выверить, что физически
  это grab/release, прежде чем гонять.
- ⚠️ Финальный `strong` release обязателен (надёжная укладка), иначе полка может не сесть.
- ⚠️ Таймаут на каждое движение лотка; неожиданный концевик = СТОП (железное правило).
- ⚠️ Перед движением лотка EN1/EN2 (25/26) → LOW; замки удерживать 500 при простое.
- Один прогон = одна операция, с присутствием Романа.
